Oscar

OSC control for broadcast video equipment

15 Feb 2016, 22:39
Development
Python / RabbitMQ / Celery / Gevent
http://github.com/ragnarula/oscar

Oscar started life as a desktop C++ called OSCBridge. The idea was simple, Set up an OSC server, establish a TCP connection to a device such as a Barco ScreenPRO-II series switcher, and be able to forward messages recieved at the OSC server to the switcher. Devices such as the barco often come with their own set of commands that can be initiated via telnet. The idea wasn’t to build an extra abstraction over the available commands, but simply to provide a more convenient interface.

The original implementation did what it was meant to but created a large budren on the developer (me!) to package the software for multiple platforms, operating systems etc etc… As well as headaches with linking to and shipping Qt libraries in a way which didn’t contradict their license terms. Eventually I came to the conclusion that it was time to change the architecture to one which didn’t depend on the user’s platform. Oscar was born.

Oscar is developed in python and is designed to be run as embedded software. This means I no longer have to worry about developing for multiple platforms, and instead offer the solution as a hardware device, or at the very least, a virtual appliance. In either case, I can specify the environment as part of the software. Development was done with a Raspberry Pi which was able to handle the workload without breaking a sweat.

Oscar also now employs an event driven, message passing design. Messages are recieved at the OSC interface and passed to seperate routines running in green threads for asyncronous processing, freeing up the OSC server to recieve more messages. Failures from malformed messages occur fast and don’t bring down the entire system. New threads are only created when establishing new connections to newly input device definitions, unlike the old system which created a new thread to process each message.

Overall, it is more efficient, more ressilient to failure and carries a much lower burden from variations in deployment environments.

Oscar was tested on a three day event in September 2015, where it was able to route messages from two Qlab controllers to nine Playback Pro machines without a hitch.
